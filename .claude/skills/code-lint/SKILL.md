# code-lint

このプロジェクトのソース（C++中心、HLSLも対象）を対象に、誤字脱字・インデントずれ・命名規約違反を検出し、各指摘に対して **報告＋修正提案** を出す。

## 範囲の決定

引数を見て範囲を決める:

1. **引数なし** — `git diff` の差分（ステージ済み + 未ステージ）を対象。
2. **コミット参照**（`HEAD~3` / `master..HEAD` / SHA など） — `git diff <参照>` の差分を対象。
3. **ファイル/フォルダパス** — そのパス配下の対象拡張子ファイルを直接読む。
4. **複数引数** — 順に解釈（フォルダ + コミット参照の組み合わせも可）。

対象拡張子: `.h` / `.hpp` / `.cpp` / `.hlsli` / `.hlsl`

`Project/externals/` 配下や `Debug/` `Temp/` などの自動生成物は対象外にする。

## チェック項目

### 1. 誤字脱字

- 英単語のtypo（`recieve` / `lenght` / `seperate` / `occured` / `referer` 等の頻出ミス、識別子・コメント・文字列リテラルのいずれも対象）
- 日本語コメントの誤変換・脱字（明らかな範囲のみ。曖昧なものは指摘しない）
- Doxygenタグの誤り（`@params` / `@returns` 等）

### 2. インデント

- タブとスペースの混在（このプロジェクトはタブ/スペースどちらか統一されているはず — 周辺ファイルの主流に合わせる）
- ブロック深さに対する不一致
- 行頭の揃っていないケース（同一ブロック内で揃っていない）
- 末尾の空白文字

### 3. 命名規約（`Project/CLAUDE.md` を参照）

- **クラス名**: PascalCase でない（`spriteManager` / `sprite_manager` 等）
- **メンバ変数**: 末尾アンダースコアが付いていない（`position` → `position_`）
- **関数名**: PascalCase でない、または `get_xxx` / `set_xxx` のスネーク形になっている
- **意味のない名前**: `temp` / `tmp` / `hoge` / `aaa` / `test` / `data1` などローカルでも意図不明なもの
- **ハンガリアン残骸**: `m_` / `g_` / `p_` プレフィックスはこのプロジェクトでは使わない方針
- **マジックな略語**: `mgr` / `cfg` / `ctx` 等が文脈なく使われている（ManagerやContextなど明示する）

## 手順

1. 引数を解釈して対象ファイルを確定する。`git diff` 経由なら変更行のみを優先的に見る（前後数行は文脈のため読む）。
2. 対象ファイルを読み、上記3カテゴリで指摘を集める。
3. 周囲の既存コードを必ず参照し、**そのファイル/フォルダの主流の流儀**に従って判定する（プロジェクト全体で統一されていない場合に局所的な揃え方を尊重）。
4. 結果を以下のフォーマットで出力する。

## 出力フォーマット

```
## code-lint 結果

対象: <範囲の説明>
件数: 誤字 X / インデント Y / 命名 Z

### 誤字脱字
- [path/to/file.cpp:42](path/to/file.cpp#L42) — `recieve` → `receive`
  修正提案: <該当行の before/after を短く>

### インデント
- [path/to/file.h:10-15](path/to/file.h#L10-L15) — タブとスペース混在
  修正提案: <方針>

### 命名
- [path/to/file.h:88](path/to/file.h#L88) — メンバ変数 `position` に末尾アンダースコアなし
  修正提案: `position_` にリネーム（参照箇所 N 件、要確認）
```

各指摘ごとに **修正の Edit 案を提示するが、自動では適用しない**。ユーザーが「これとこれを直して」と指示したらそのときに Edit を実行する。

## やらないこと

- 大規模リネーム（命名違反が多数ある場合は件数だけ報告し、ユーザーに範囲を確認する）
- 機械的でない判断（コメントの言い回しの好み、関数の責務分割など）
- 自動コミット
- 対象外ファイル（`externals/` / 生成物）への指摘

## 偽陽性を避ける配慮

- ライブラリ由来の識別子（`HRESULT`, `DXGI_*` 等）には触れない
- 既存ファイル全体で統一されている流儀（例: そのファイルだけスネークケース）はプロジェクト規約と齟齬があっても局所では尊重し、「ファイル単位で違っている」事実だけ報告する
- typo候補が固有名詞・専門用語の可能性がある場合は「不確実」マークを付ける
