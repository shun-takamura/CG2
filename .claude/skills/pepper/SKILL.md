# pepper — 最適化アドバイスシステム（PEPPER Step7）

PEPPER が出した `profile.log`（1秒ウィンドウ集計）を解析し、**ボトルネックを特定して具体的な最適化案を提示する**。
計画書 [`P.E.P.P.E.R..md`](../../../P.E.P.P.E.R..md) §5/§7 の Step7。**v1 はあくまで「提案まで」**＝採否と実装はユーザーが判断する（自動修正→PR は将来拡張）。
**着手前にこの SKILL 全体を読むこと。**

## 参照データ（このスキルの核）

- **解析ツール**: [`Project/tools/Python/analyze_profile.py`](../../../Project/tools/Python/analyze_profile.py)
  — profile.log を読み、区間別 CPU/GPU 時間・呼出回数・µs/回・予算%・カウンタ・ゲージ・無駄ヒントを集計。`--json` で機械可読出力。**このスキルの入口。**
- **ログ置き場**: `Project/Logs/<セッション>/profile.log`（最新セッションは analyze_profile.py が自動検出）。
- **最適化の指針（提案の根拠）**: [`Project/CLAUDE.md`](../../../Project/CLAUDE.md) の「パフォーマンスについて」
  — 描画コール削減・定数バッファのまとめ・SRV のバッチ化・ヒープ確保の削減・SIMD・`std::move`・コンパイル時計算。**提案はこの語彙に紐付ける。**
- **計装の実体**: 各所の `PEPPER_SCOPE("名前")`。区間名でコードを grep すれば該当箇所に飛べる（`file:line` も JSON に入っている）。

## 手順

1. **データを取る。** 最新セッションを解析:
   ```
   python Project/tools/Python/analyze_profile.py --json
   ```
   - セッションやFPS基準を変えるなら `<session_dir>` / `--fps 240` / `--top 15` を付ける。
   - 人が眺める用のレポートは `--json` 無しで実行（CPU/GPU 最遅 TOP・効率チェック・ゲージが表で出る）。
   - **profile.log が無い** = まだ計測していない。ユーザーに「Debug/Development ビルドでゲームを少し動かして profile.log を作って」と促す（このスキルは計測済みログが前提）。
2. **全体像を掴む。** JSON の `avg_fps` と `fps_budget_ms` を見て、そもそも予算内か確認。
   - ⚠️ 本エンジンは **VSync 律速**（実測 約229fps）。定常では CPU が VSync を待つ遊び時間（`GpuWait`/`Present`）が大半で、「定常ボトルネックなし」が正常な結論になり得る。**無理に犯人を作らない。** その場合は「予算に余裕あり。最適化より機能追加余地の報告」を返す。
3. **容疑区間を絞る。** 次の3軸で見る（JSON にすべて揃っている）:
   - **時間が重い**: `cpu_per_frame` / `gpu_per_frame` の TopN（`dominant` で CPU/GPU 律速を区別）。
   - **回数で重い**: `calls_per_frame` 大 × `cpu_per_call_us` 小 → 「1回は安いが回数で効く」＝まとめ/キャッシュ候補。
   - **既出ヒント**: JSON の `hints`（DrawCall 過多・呼出過多）を起点にする。
   - ⚠️ **親子に注意**: `depth=0/1` は子を含む合計（親）。**最適化は深い（葉）区間の重い所から**。親の数字だけ見て親を直そうとしない（self-time は未対応＝§後述）。
4. **コードを読む。** 容疑区間の `file:line` を Read（必要なら区間名で Grep して呼び出し元・ループ・確保を確認）。「なぜ重いか」を**実コードで**裏取りする。推測で提案しない。
5. **施策に変換する。** 下記「ボトルネック種別 → 施策マッピング」で CLAUDE.md 指針に紐付ける。
6. **提案する。** 下記「提案フォーマット」でランク付け。**採否はユーザー。** 勝手に大規模書き換えはしない（CLAUDE.md ルール）。

## ボトルネック種別 → 施策マッピング

| 観測（JSON の何を見るか） | 疑い | CLAUDE.md 指針の施策 |
|---|---|---|
| `counter=DrawCall` 平均が多い / `hints` に DrawCall 警告 | 描画コール過多 | **描画コール削減**：同マテリアル/メッシュのバッチ化・インスタンシング・テクスチャアトラス化 |
| `dominant=GPU` で `gpu_per_frame` 大 | GPU 律速（塗り/シェーダ/オーバードロー） | シェーダ簡略化・解像度/RT 見直し・オーバードロー削減・不要パス削除（PostEffect 等） |
| `calls_per_frame` 大 × `cpu_per_call_us` 小 | 細かい呼び出しの累積 | **呼び出しのまとめ／キャッシュ**・ループ外への巻き上げ・**定数バッファのまとめ**・**SRV のバッチ化** |
| `cpu_per_call_us` 大（1回が重い） | 重い単発処理（衝突全探索・スキニング・整列等） | アルゴリズム/データ構造の見直し・**SIMD**・空間分割・**コンパイル時計算(constexpr)** |
| 毎フレーム生成系（パーティクル/一時オブジェクト）が重い | 動的確保の多発 | **ヒープ確保の削減**：プール化・`reserve`・**`std::move`**・使い回し |
| `gauge=RAM_WS_MB / VRAM_MB` が単調増加 | リーク/確保しっぱなし | 寿命管理・解放漏れ・キャッシュ上限・アセット重複ロード（`Tex_Count`/`Model_Count` と突合） |

- **CPU 表に `◀GPU` 印**が付く区間は、CPU 表に出ていても実際は GPU 律速 → CPU 側を弄っても効かない。施策は GPU 側へ。
- 「**時間ランキング＝重い処理**」「**効率チェック＝回数で効く無駄**」は別物。両輪で見る（analyze_profile.py の注記どおり）。

## 提案フォーマット

ボトルネックごとに、**根拠（実測値＋コード）→ 施策 → 想定効果/コスト** を簡潔に。確信度順に並べる。

```
【1】 <区間名>  [file:line]  — <CPU/GPU> <ms/f>（予算の <%>） / 呼出 <n>/f（<µs/回>）
  原因  : <コードのどこがなぜ重いか。実コードでの裏取り>
  施策  : <CLAUDE.md 指針の具体策。1〜3案、低コスト順>
  効果  : <定量目安。例「DrawCall 200→1 でCPU約Xms削減見込み」>
  コスト: <改修規模/リスク（描画結果が変わる等）>
```

- 効果が読めないものは「要計測」と正直に書く。**盛らない。**
- 1件直すごとに再計測（profile.log を取り直して再解析）して効果を確認する運用を勧める。

## 注意・スコープ

- **v1 は提案まで。** 実装は別途ユーザーの指示で（その際 Edit/Write は CLAUDE.md の事前通告ルールに従う）。
- **計測に触らない読み取り専用**：解析はゲーム挙動（乱数・dt）に影響しない＝SUNDAY の決定論リプレイと両立。
- **VSync 律速を忘れない**：定常で犯人が見つからないのは正常。負荷シーン（敵大量・必殺技発動中など）のログを取らせると差が出やすい。
- **self-time（排他時間）は未対応**（計画 §C）。今は親が子を含む合計なので、葉まで `depth` を辿って判断する。将来 Profiler に親リンクを足せば「親時間−子合計」で自己時間が出せ、提案が鋭くなる。
- **将来拡張**：採用された施策を「修正→ビルド→PR」まで自動化（SUNDAY の自動修正トラックと共通の `/prompt` 経路）。今はやらない。
